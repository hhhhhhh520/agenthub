## 发现的问题

### ISSUE-001: ESLint 错误 - 未使用的变量 (Severity: Medium)

**文件**: `src/app/api/sessions/[id]/chat/route.ts`
**位置**: 多处
**问题**: 9 个 `@typescript-eslint/no-unused-vars` 错误
**影响**: 代码质量下降，可能遗漏重要逻辑

**示例**:
- `cliSessionId` 变量在多处声明但未使用
- `task` 变量在某些循环中未完全使用

---

### ISSUE-002: React Hooks 规则违规 (Severity: High)

**文件**: 多个组件文件
**位置**: useEffect 中直接调用 setState
**问题**: 5 个 `react-hooks/set-state-in-effect` 警告

**受影响文件**:
- `src/components/agent-panel.tsx` (line 57)
- `src/components/create-group-dialog.tsx` (line 43, 55)
- `src/components/create-agent-dialog.tsx`
- `src/components/provider-import-dialog.tsx`
- `src/lib/hooks/use-sessions.ts`

**问题分析**:
在 useEffect 中直接调用 setState 可能导致不必要的重渲染循环。虽然当前代码逻辑上没有形成死循环，但违反了 React 最佳实践。

**示例代码** (`agent-panel.tsx:57`):
```typescript
useEffect(() => {
  loadAgents()  // loadAgents 内部调用 setAgents
}, [sessionId])  // 缺少 loadAgents 依赖
```

---

### ISSUE-003: React Hooks 依赖缺失 (Severity: Medium)

**文件**: `src/components/agent-panel.tsx`
**位置**: line 57, 60-70
**问题**: `react-hooks/exhaustive-deps` 和 `react-hooks/immutability` 警告

**分析**:
- `loadAgents` 函数未加入 useEffect 依赖数组
- 可能导致 sessionId 变化时无法正确响应

---

### ISSUE-005: Turbopack 构建警告 (Severity: Low)

**文件**: `src/lib/workspace.ts`
**问题**: 动态文件系统操作导致 Turbopack 警告
**警告内容**:
```
Encountered unexpected file in NFT list
A file was traced that indicates that the whole project was traced unintentionally
```

**原因分析**:
`workspace.ts` 使用了 `process.cwd()` 和动态 `fs` 操作，Turbopack 无法静态分析依赖范围。

### ISSUE-007: API 错误处理不完善 (Severity: Medium)

**文件**: `src/app/api/sessions/[id]/chat/route.ts`
**位置**: 多处 catch 块
**问题**: 部分 catch 块静默失败，仅打印 console.error

**示例**:
```typescript
} catch (err) {
  sendEvent({ agentId: agent.name, type: 'error', content: `执行失败: ${err instanceof Error ? err.message : String(err)}` })
}
```

虽然发送了错误事件，但没有记录完整错误栈，调试困难。

---

### ISSUE-008: 潜在的类型安全问题 (Severity: Medium)

**文件**: `src/lib/orchestrator/index.ts`
**位置**: parseJSON 函数
**问题**: JSON 解析失败时的错误消息截断可能丢失关键信息

```typescript
throw new Error(`Failed to parse JSON from: ${text.slice(0, 200)}`)
```

当 LLM 返回大量文本时，截断到 200 字符可能导致无法定位实际问题。

---

### ISSUE-009: Session 锁机制可能泄漏 (Severity: Medium)

**文件**: `src/app/api/sessions/[id]/chat/route.ts`
**位置**: sessionLocks Map 管理
**问题**: 在异常情况下 sessionLocks 可能未正确清理

**代码分析**:
```typescript
// Clean up lock if this is still the current holder
if (sessionLocks.get(sessionId) === current) sessionLocks.delete(sessionId)
```

如果 finally 块执行前发生崩溃，锁可能残留。

---

### ISSUE-010: Prisma 模型缺少索引 (Severity: Low)

**文件**: `prisma/schema.prisma`
**问题**: 高频查询字段缺少索引优化

**建议添加索引**:
- `Session.updatedAt` (用于排序会话列表)
- `Message.sessionId + createdAt` (用于加载消息历史)
- `Task.sessionId + status` (用于任务状态查询)

---

### ISSUE-011: Provider API Key 安全风险 (Severity: High)

**文件**: `src/app/api/providers/route.ts`
**位置**: GET 端点返回 apiKey
**问题**: API 直接返回 apiKey 字段，可能泄露敏感信息

**代码**:
```typescript
return NextResponse.json(unique)  // unique 包含 apiKey 字段
```

**建议**: 不应将 apiKey 返回给前端，应仅返回配置元信息。

---

### ISSUE-012: 环境变量默认值不安全 (Severity: Medium)

**文件**: `src/lib/db.ts`
**位置**: DATABASE_URL 默认值
**问题**: 默认使用 `file:./dev.db`，在生产环境可能意外使用开发数据库

```typescript
const adapter = new PrismaLibSql({
  url: process.env.DATABASE_URL || 'file:./dev.db',
})
```

## 总结

1. **测试覆盖缺失** - 这是最严重的问题，无法验证功能的正确性

项目可以正常构建和运行 TypeScript 检查通过，说明基础代码质量尚可。建议优先添加测试覆盖，然后逐步修复其他问题。
