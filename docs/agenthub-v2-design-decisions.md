# AgentHub v2 设计决策记录

> 讨论时间: 2026-05-20 | 状态: 讨论中，待完善

---

## 已确定的决策（12 项）

### 1. 执行层：混合模式

- 分析/讨论用 LLM API（Vercel AI SDK 直连，无进程创建）
- 代码任务用 Claude Code CLI（真实文件操作）
- Orchestrator 的 3 次分析调用从 3 个 CLI 进程降到 0 个
- **原因**: 全 CLI 模式每次对话至少 3 个进程，token 浪费严重

### 2. Agent 池：全局共享，预设模板

- Agent 预定义，所有会话复用
- 提供常见 Agent 模板（架构师、前端工程师、后端工程师、测试工程师等）
- 用户只需从 CC-Switch 导入的服务商中选择合适的模型
- **原因**: Agent 有持久身份，跨会话复用，模型配置稳定可控

### 3. 模型绑定在 Agent 上

- 不是按任务选模型，而是按 Agent 选模型
- 每个 Agent 配置自己的 LLM（如架构师用 claude-opus，测试用 deepseek）
- **原因**: 不同 Agent 能力需求不同，按 Agent 配置更精确

### 4. 服务商来源：CC-Switch 导入 + 手动添加

- 从 CC-Switch 一键导入服务商配置（API 端点、模型、Key）
- 也支持手动添加服务商
- **原因**: 用户已经在 CC-Switch 里配好了，不用重复配置

### 5. Orchestrator：系统级组件

- 不进 Agent 池，用户不能删除或修改
- 只负责选人 + 拆任务，不造人
- 选不到合适 Agent → 告知用户去创建
- **原因**: Orchestrator 是系统核心调度器，不是普通 Agent

### 6. 群聊结构：Orchestrator + Agent 在同一个对话里

- 不是"先和 Orchestrator 聊，再创建独立群聊"
- 而是一个对话里有 Orchestrator + 多个 Agent，类似飞书群聊
- **原因**: 官方课题要求"一个对话中包含多个 Agent"

### 7. Agent 进群流程

- Orchestrator 先自动选择合适的 Agent → 展示给用户
- 用户可以增减 Agent → 用户确认 → 建群
- 建群操作由用户执行
- **原因**: 用户掌控 Agent 组合，Orchestrator 只推荐

### 8. 回复顺序：分阶段

- **对齐阶段**: Orchestrator 指定顺序，每个 Agent 轮流和用户确认
- **执行阶段**: 按任务依赖，无依赖的并行，有依赖的等前置完成
- **原因**: 对齐阶段有序不混乱，执行阶段高效能并行

### 9. 对齐流程

合并了需求确认 + 技术方案 + 任务拆解，一次对齐完成三件事：

```
Orchestrator: @产品经理 请确认需求
产品经理: 个人 TODO 应用，功能：增删改查 + 分类 + 搜索，对吗？
用户: 对，加上优先级标记

Orchestrator: @架构师 请出技术方案 + 任务拆解
架构师:
  技术方案：Vue 3 + Pinia + Node.js + Express + SQLite
  任务拆解：
  1. 搭建项目脚手架（前端工程师）— 无依赖
  2. 设计数据模型（后端工程师）— 无依赖
  3. 实现 API 接口（后端工程师）— 依赖 2
  4. 实现前端页面（前端工程师）— 依赖 1、3
  5. 编写测试用例（测试工程师）— 依赖 3、4

Orchestrator: 其他 Agent 有问题吗？
前端工程师: UI 风格有偏好吗？
用户: 简洁一点
后端工程师: （沉默）
测试工程师: （沉默）

Orchestrator: 对齐完成，进入执行阶段
```

| 对齐内容 | 负责人 | 对应颗粒度管理 |
|----------|--------|--------------|
| 需求确认 | 产品经理 | 项目分解 |
| 技术方案 | 架构师 | 任务定义 + 颗粒度调整 |
| 任务拆解 + 分配 | 架构师 | 资源分配 |

### 10. 任务颗粒度

- 一个任务 = 一个 Agent = 一次或少数几次 LLM 调用 = 一个明确的工件产出
- **原因**: 太大 Agent 做不完，太小协调开销大于收益

### 11. 通信模式：聊天对齐 + 工件执行

- 对齐阶段用聊天（讨论、确认）
- 执行阶段用工件（结构化文档/文件传递）
- 工件 = Agent 产出的结构化文档，如 API 契约、代码文件、测试报告
- Agent 之间通过工件协作，保证产出一致性
- **原因**: 聊天灵活适合讨论，工件精确适合执行

### 12. 每个 Agent 可选择执行平台

- 用户创建/编辑 Agent 时，可选择该 Agent 的执行平台：Claude Code / Codex / LLM API
- Orchestrator 固定使用 LLM API（系统组件，做分析/调度，不需要文件操作）
- 典型配置：
  - 架构师 → LLM API（产出设计文档、API 契约，不写代码）
  - 产品分析师 → LLM API（需求分析，产出需求文档）
  - 前端工程师 → Claude Code CLI（写代码文件）
  - 后端工程师 → Claude Code CLI（写代码文件）
  - 测试工程师 → Claude Code CLI（写测试文件）
- 不同平台的工件存储方式不同：
  - LLM API Agent → 产出文本工件 → 存数据库
  - CLI Agent → 产出文件工件 → 存文件系统，数据库存路径
- **原因**: 统一适配器层的核心价值——不同 Agent 按职责选择最合适的执行平台

### 13. 群聊消息可见性：全员可见

- UI 层：所有消息对所有 Agent 和用户可见，模拟飞书/微信群聊体验
- Agent 上下文：完整聊天历史传给每个 Agent，不做数据层过滤
- Orchestrator 调度：内部可定向分派任务给子 Agent，但指令和子 Agent 回复都广播到群聊界面
- 噪音缓解：通过 Orchestrator 对齐流程（明确任务分配）+ 阶段控制来解决，不靠隐藏消息
- **原因**: 课题要求"IM 聊天式"群聊体验；Orchestrator 需要看到所有子 Agent 产出才能聚合和协调；全员可见形成"多专家现场讨论"的协作氛围，避免信息孤岛。核心原则：执行可以定向，呈现必须全员可见

### 14. 会话恢复与上下文管理

**双模式 CLI 进程管理**（参考 cc-connect + multica 的做法）：

| 场景 | 模式 | 实现方式 |
|------|------|----------|
| 当前会话活跃时 | 长驻进程 | `--input-format stream-json` + stdin/stdout JSON 管道，进程保持活着，消息实时发送 |
| 会话恢复时 | 一次性进程 + sessionID 恢复 | `--resume sessionID`，spawn 新进程传入上次的 sessionID，Claude Code 内部恢复上下文 |

**sessionID 持久化**：
- CLI Agent 每次执行后从输出中提取 sessionID，存入数据库（Agent 表的 `sessionId` 字段）
- 下次执行时传入 `--resume sessionID`，Claude Code 内部维护上下文，不需要重新传历史
- 失败回退：如果 `--resume` 失败（session 不存在），清空 sessionID，用新会话

**工件持久化**：
- CLI Agent 的工作目录从 `/tmp` 改到项目内固定位置 `workspaces/{sessionId}/`，关机不丢失

**任务恢复**：
- 用户重新打开会话时，检查有没有 `in_progress` 状态的任务，提示用户「上次有未完成的任务，是否继续？」

**原因**: 参考 cc-connect（长驻进程 + stdin/stdout 管道）和 multica（一次性进程 + `--resume sessionID`）的成熟方案，兼顾实时交互和断线恢复

### 15. 消息操作交互方式

| 端 | 触发方式 | 优先级 |
|---|----------|--------|
| Web 端 | 消息卡片上放小图标（如 `⋯` 或 `⋮`），点击弹出操作菜单（回复、引用、重新生成、复制） | P1 |
| 桌面客户端 | 鼠标右键弹出上下文菜单 | P2 |
| 手机端 | 长按弹出操作菜单 | P2 |

- 回复机制：点击「回复」→ 输入框上方显示引用预览条（发送者 + 前 50 字）→ 用户输入补充内容 → 发送时带 `replyToId` 字段 → 前端气泡下方展示被引用消息缩略
- 数据模型：Message 表加可选 `replyToId` 字段，关联自身
- 查消息时 `include: { replyTo }` 带上被引用消息
- 被引用消息删除时显示「原消息已删除」
- Agent 上下文不需要特殊处理，会话完整聊天历史已包含所有消息
- 重新生成：复用 chat API，传 `regenerate: messageId`，后端重新执行，前端替换对应消息
- **原因**: 桌面端/手机端为 P2，Web 端优先实现

### 16. 消息内容格式：纯文本存储 + 读取时解析

- 数据库只存 `rawContent`（Agent 原始纯文本输出）
- 读取 API 时实时解析 `rawContent`，生成结构化 `content`，返回 `{ rawContent, content }` 给前端
- 解析逻辑集中在后端 API 层，前端只负责按类型渲染
- 解析失败时 `content` 降级为空，前端回退显示 `rawContent` 原始文本
- **原因**: 20 天比赛项目解析逻辑会频繁变动，读取时解析避免"改了解析器 → 旧数据 stale → 要跑迁移"的问题；解析只是正则提取，性能开销极小

**结构化 content 定义**：

```typescript
interface MessageContent {
  text?: string                    // 纯文本部分
  codeBlocks?: Array<{             // 代码块
    language: string
    code: string
  }>
  artifacts?: Array<{              // 产物卡片
    type: 'web-preview' | 'diff' | 'file' | 'deploy'
    data: Record<string, unknown>
  }>
}
```

### 17. 产物内联

| 产物 | 渲染方式 | 数据来源 | 优先级 |
|------|----------|----------|--------|
| 代码块 | 语法高亮 + 复制按钮 | Agent 回复中的 markdown 代码块 | P1 |
| 网页预览 | iframe srcdoc + sandbox | Agent 回复中的 HTML | P1 |
| 文件附件 | 卡片（文件名+大小+下载） | 工作目录文件 | P1 |
| Diff 视图 | Monaco DiffEditor + Accept/Reject | Agent 回复中的 diff 文本 | P1 |
| 部署状态 | 进度条 + URL | 部署 API | P2 |
| PPT 浏览 | iframe / PDF.js | 生成的文件 | P2 |

**网页预览卡片**：
- 用 `srcdoc` + iframe 渲染，不需要部署即可实时预览
- 必须加 `sandbox="allow-scripts"`，不能加 `allow-same-origin`（防止恶意脚本访问主页面）
- HTML 有语法错误时降级显示提示"预览可能不完整"
- artifacts 数据：`{ type: "web-preview", data: { html: "..." } }`

**文件附件卡片**：
- 文件存在 `workspaces/{sessionId}/` 目录
- 文件用 UUID 命名物理文件，数据库存原名（防路径遍历攻击）
- 下载 API 需校验 sessionId 归属（权限隔离）
- artifacts 数据：`{ type: "file", data: { fileName, fileSize, downloadUrl } }`

**Diff 视图卡片**：
- 优先实现"Agent 输出提取"：Agent 在回复中直接给 diff 文本（` ```diff ` 代码块），不依赖文件系统对比
- Monaco DiffEditor 设 `readOnly: true`，高度 300px，点击展开全屏 Modal
- Accept → 后端写文件到工作目录；Reject → 丢弃
- Accept 前检查文件是否被用户手动修改过，不一致则提示"文件已被修改，是否覆盖？"
- Accept/Reject 后前端卡片状态更新
- artifacts 数据：`{ type: "diff", data: { filePath, original, modified } }`

### 18. 代码冲突处理：声明预防 + 合并审计

两层机制，砍掉实时监控，复用 Diff 卡片：

**声明阶段（预防）**：
- Orchestrator 拆任务时输出每个任务的 `declared_files`（预期修改的文件集合）
- 文件重叠的任务排为串行依赖，不重叠的可并行
- 闭合规则：声明内文件 Agent 自由修改；未声明的文件如果被修改，合并时视为"越界修改"，必须走用户确认

**执行阶段（隔离）**：
- 每个 Agent 在隔离工作区执行：`workspaces/{sessionId}/task-{taskId}/`
- 文件系统副本，不需要容器或沙箱

**合并阶段（审计）**：
- 对每个 task 工作区运行 `git diff --name-only` 或快照对比，拿到实际改动文件列表
- 与 `declared_files` 取差集：一致的直接合并，不一致的为"额外修改"
- 额外修改生成 Diff 卡片推送到聊天流，让用户 Accept/Reject
- 不需要解析 diff 内容，只做文件级别的对比

**原因**: 通过 Diff 卡片将冲突处理可视化，契合 AgentHub "聊天式协作"的产品基因；两层机制比三层更务实，20 天比赛可交付

### 19. 用户自建 Agent：对话式创建 + 表单创建

两种入口并存，底层共用同一套配置模型：

| 入口 | 定位 | 适合谁 |
|------|------|--------|
| 对话式创建 | 主要方式（课题要求） | 普通用户，自然语言描述需求 |
| 表单创建 | 辅助方式（快捷入口） | 高级用户，直接填配置 |

**对话式创建流程**：
1. 用户告诉 Orchestrator："帮我创建一个擅长 React 的前端工程师"
2. Orchestrator 生成配置：名称、专长、System Prompt、执行平台、模型、工具集
3. 用户确认或提出修改（如"模型用 deepseek"）
4. 确认后 Agent 创建完成，出现在联系人列表

**表单创建**：Agent 面板提供「快速创建」按钮，填完配置直接生成，跳过 Orchestrator 对话。

**配置项**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 名称 | ✅ | 中文角色名，如"前端工程师" |
| 专长描述 | ✅ | 如"擅长 React、TypeScript、CSS" |
| System Prompt | ✅ | Agent 的行为规范和角色定义 |
| 执行平台 | ✅ | LLM API / Claude Code CLI |
| 模型 | ✅ | 根据执行平台选择 |
| 工具集 | ✅ | Orchestrator 根据描述自动推荐，用户可调整 |
| 头像 | ❌ | 默认首字母 + 颜色，可自定义 |
| 能力标签 | ❌ | Badge 展示 |

**工具集管理**：
- 后端维护工具集预设映射：`"代码读写" → [Read, Write, Edit]`、`"命令执行" → [Bash]` 等
- Orchestrator 根据用户描述组合工具集
- Agent 表增加 `tools: string[]` 字段存储
- 执行时适配器根据 `tools` 加载对应工具实现（MCP tools 或 function calling）

**数据模型**：Agent 表加 `isPreset: boolean` 区分预设和用户自建。用户自建 Agent 全局可用，不绑定会话。

**原因**: 课题明确要求"对话式创建，设定 System Prompt + 工具集"；表单创建作为辅助入口满足高级用户个性化需求，不增加维护成本

### 20. Agent 展示：联系人样式

课题要求"每个 Agent 在聊天列表中显示为独立的联系人，有头像、名称、能力标签"。

| 属性 | 实现 | 优先级 |
|------|------|--------|
| 头像 | 首字母 + 专属背景色（预设 Agent 按角色分配固定色，自建 Agent 按名称哈希取色），不用 emoji 避免跨平台渲染差异 | P1 |
| 名称 | 中文角色名（如"前端工程师"），支持用户自定义 | P1 |
| 能力标签 | Badge 展示（如 `React` `TypeScript`），创建时配置 | P1 |
| 状态指示 | Agent 面板：圆点颜色（空闲灰、工作中绿、完成蓝、出错红）；消息气泡中可选（P2） | P1/P2 |
| 平台标识 | Agent 面板卡片 + 对话初始消息中展示；消息气泡和会话列表不展示，避免冗余 | P1 |
| 主题色 | 每个 Agent 分配 `accentColor`，存数据库，用于头像背景、气泡左侧色条、名称高亮 | P1 |

**预设色盘**：8-12 种高区分度颜色（靛蓝、翠绿、琥珀、红、紫等），按序或哈希分配给 Agent，保证群聊中相邻 Agent 颜色不相似。

**展示位置**：
- 右侧 Agent 面板：卡片列表，每个 Agent 一张卡片
- 聊天消息气泡：左侧显示头像 + 名称，气泡左侧有主题色色条
- 会话列表：群聊会话显示参与的 Agent 头像拼图

**状态同步**：后端维护 `idle / working / done / error` 四个状态，由 Orchestrator 在任务生命周期中更新。

**原因**: 课题明确要求 Agent 作为独立联系人展示；主题色方案保证群聊中视觉区分度；平台标识克制展示避免噪音

## Orchestrator 群聊角色

- **主持人**: 控制阶段切换（对齐 → 执行）
- **监督者**: 沉默观察，站在上帝视角看整体方向
- **纠偏者**: 发现跑偏时 @Agent 纠正
- 正常运行时沉默，只在关键节点介入

### 21. AI 协作规范（30% 权重）

考察的是**开发者在开发过程中和 AI 的协作方式**，不是项目内部 Agent 之间的协作。

| 层 | 含义 | 落地方式 |
|---|------|----------|
| Spec | 给 AI 写的任务规范 | 设计文档 + 实现计划（已有） |
| Skill | 沉淀的 AI 协作技巧 | 开发完成后从历史对话中 AI 分析总结，产出 `docs/ai-collaboration.md` |
| Rules | 和 AI 之间的协作规则 | CLAUDE.md、AGENTS.md（已有） |

**原因**: 先开发再总结，比提前编更有说服力；答辩时展示 Spec + Skill + Rules 三层协作规范

### 22. 接入平台：Claude Code + OpenCode

| 平台 | 启动命令 | Session 恢复 | 用途 |
|------|----------|-------------|------|
| Claude Code CLI | `claude -p --output-format stream-json --input-format stream-json` | `--resume sessionID` | 代码任务 |
| OpenCode CLI | `opencode run --format json` | `--session sessionID` | 代码任务 |
| LLM API | Vercel AI SDK | 不适用（无状态） | 分析/讨论 |

OpenCode 集成难度与 Claude Code 基本一致，cc-connect 和 multica 都有现成参考代码。
- 权限自动批准：环境变量 `OPENCODE_PERMISSION={"*":"allow"}`
- Prompt 传入：命令行末尾参数
- 输出格式：JSON 流（text/tool_use/step_finish/error）

**原因**: 课题要求至少接入 2 个主流 Agent 平台；OpenCode 开源免费，已下载，有成熟参考实现

## 上下文管理

- **自动上下文**: 聊天历史自动作为上下文传递给 Agent（会话级），会话内不压缩，保持完整历史
- **长期上下文**: 用户手动 pin 关键消息，绑定到 Agent（Agent 级，跨会话可见）
- Agent 进入新会话时，上下文 = 当前会话完整消息历史 + Agent 级 pin 的消息
- **多轮迭代修改**: 用户可以反复修改 Agent 的产出，每一轮 Agent 都能看到之前所有对话和自己之前的产出
  - 示例: Agent 写了代码 → 用户说"改背景色" → Agent 在之前代码基础上修改 → 用户说"再加个按钮" → Agent 继续修改
  - 要求: 每轮对话完整保留，Agent 的上下文包含自己之前产出的工件（代码文件等）

## 数据模型要点

- Agent 全局预定义，通过 SessionMember 中间表与 Session 建立多对多关系
- 一个 Agent 可以同时在多个群聊里
- 上下文按会话隔离，Agent 级 pin 跨会话共享
- Task 有依赖关系，分配给 Agent
- 工件存储按 Agent 平台区分：LLM Agent 存数据库，CLI Agent 存文件系统

## 失败处理策略

### 错误分类与重试策略

盲目重试会浪费 token。需要按错误类型决定是否重试：

| 错误类型 | 示例 | 策略 |
|----------|------|------|
| 可重试 | 限流（429）、网络错误、超时 | 重试 + 指数退避 |
| 不可重试 | API Key 错误、权限不足、上下文超长 | 直接终止，不浪费重试机会 |
| 半可重试 | 返回格式错误（非 JSON） | 重试时调整 prompt（如要求"只输出 JSON"） |

### 降级前的能力检查

降级到备用模型前，必须检查：

- 备用模型是否支持同样的 function calling / tool use
- 上下文窗口是否够大（上游已积累 150K token，不能降到 128K 窗口的模型）
- 输出格式约束能否满足（JSON 跟随能力）

能力不满足 → 跳过降级，直接通知用户。

### 质量自动检测机制

Orchestrator 纠偏需要具体的检测手段，不能只靠"感觉"：

| 检测方式 | 适用场景 | 示例 |
|----------|---------|------|
| 产物编译/解析 | 代码类 | 前端代码跑 build / 语法检查，失败 = 质量差 |
| 断言检查 | 测试类 | 要求覆盖率 80%，结果只有 10% |
| 语义核对 | 通用 | 用小模型/规则检查产出是否符合任务描述（"要求 TODO 应用，产出里有没有增删改查？"） |
| 结构化输出校验 | JSON 类 | 要求 JSON Schema，直接验证格式 |

### 纠偏熔断器

防止 Agent 反复犯错进入死循环：

```
同一个任务 + 同一 Agent：
  纠偏超过 2 次仍不合格 → 直接终止
  → 通知用户，附带每次产出差异，方便人判断
```

### 重试时的上下文隔离

重试/重新执行时，需要清除之前失败的中间状态：

- 回滚到任务开始前的上下文快照
- 防止上一次的错误输出污染新一轮执行
- 非幂等操作标记 `unsafe_to_retry`，失败了直接交给人

### 上游失败的阻塞传播

依赖拓扑分析，只阻塞受影响的下游，不影响其他分支：

```
任务 1（脚手架）✅ 完成
任务 2（数据模型）❌ 失败
任务 3（API 接口）⏸ 阻塞（依赖任务 2）
任务 4（前端页面）✅ 可继续（依赖任务 1，不依赖任务 2）
任务 5（测试）⏸ 阻塞（依赖任务 3）

→ 汇报给用户：任务 4 可以先交付，任务 3/5 被阻塞
```

### 用户操作面板

通知用户失败时，提供可操作的选择：

| 操作 | 说明 |
|------|------|
| 重试 | 可以改参数/换模型后重试 |
| 跳过 | 标记为"人工完成"，解除下游阻塞 |
| 回滚重做 | 修改任务描述后重新分配 |
| 手动修复 | 用户直接修改产物，然后继续流水线 |

### 全链路可观测

所有失败、重试、降级、纠偏都要有结构化 trace 日志：

- 任务为什么失败？（错误类型、错误信息）
- 重试了几次？每次的错误有什么不同？
- 降级到哪个模型了？效果如何？
- Orchestrator 为什么没发现跑偏？（检测环节日志）
