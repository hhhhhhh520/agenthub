## 二、沟通风格（附原文证据）

### 2.1 "路径+动词" 模式

用户习惯直接给出文件路径，让AI自己去理解和探索：

```
"D:\projects\agenthub"看看待办
"D:\projects\agenthub\docs\课题.txt"能达到课题的要求吗
"D:\projects\web\DESIGN.md"看看这个
"D:\projects\multica"以及可以探索这个项目是怎么做的
"D:\projects\agenthub\docs\design"有没有之前改过没改好/改出新bug的
```
> 证据来源: 多个会话文件

**特征**: 不解释上下文，假设AI能读懂路径并自行探索。

### 2.2 "效果前置" 提问法

在执行前必问效果，先评估再决定：

```
"改完能达到什么样的效果"
"改完能达到什么样的效果，告诉我效果就行，不要改代码"
"修复完能达到什么样的效果"
"那方案一应该怎么实施呢，给我一份计划，包含怎么达成目标效果"
```
> 证据来源: C--Users-user 主会话（6月）

**特征**: 先问效果再决定是否执行，避免做无用功。

### 2.3 "约束条件" 前置

明确边界，防止AI越权：

```
"不要修改任何文件只需要给我一份最终的报告"
"只需要测试并报告问题就行了，不要修改任何文件"
"只要测agenthub另外两个先不要管"
"改动要到位要能达到改动的目的，同时不要对其余正常功能产生影响"
"开始实现并保证改动达到真实目的且不影响其余正常功能"
```
> 证据来源: C--Users-user 主会话（5-6月）

**特征**: 高频使用"不要"、"只需要"、"只要"等约束词。

### 2.4 "渐进式" 推进

一步一步来，不一次性下达所有指令：

```
"逐条确认，一个一个来"
"一次只讨论一个点"
"1-3先改了，改完再把剩余的告诉我"
"仔细再看看https://open-code.ai/zh/docs/tools"（重复3次，每次要求更深入）
```
> 证据来源: C--Users-user 主会话（6月）

**特征**: 控制节奏，每步都可验证。

### 2.5 "质疑验证" 模式

不盲目相信AI的输出：

```
"真的无测试文件吗，我记得之前测试过啊，你先找找，真的没有就告诉我"
"验证之前的几个其他的agent给出的报告的真实性"
"这是另一个ai搜索到的结果，你可以先核实一下信息的真实性再做判断"
"agent是预定义的吧，你仔细看看"
```
> 证据来源: C--Users-user 主会话（5-6月）

**特征**: 对AI的结论保持怀疑，要求验证。

## 三、工作流程（附原文证据）

### 3.1 典型工作流

```
┌─────────────────────────────────────────────────────────────┐
│  1. 探索: "看看项目" / "看看待办" / "仔细看看"                │
│     ↓                                                         │
│  2. 评估: "能达到什么效果？" / "还有哪些待办？"                │
│     ↓                                                         │
│  3. 决策: "做，方案是什么" / "1-3先改了" / "先不动"             │
│     ↓                                                         │
│  4. 执行: "开始实现" / "生成计划一个一个执行"                   │
│     ↓                                                         │
│  5. 验证: "测试验证" / "重启开发服务器后测试"                   │
│     ↓                                                         │
│  6. 迭代: "剩下的几个问题是什么" / "还有什么待办"               │
└─────────────────────────────────────────────────────────────┘
```

## 四、AI 协作模式

### 4.1 "指挥官" 模式

用户是决策者，AI是执行者。所有交互都指向：
1. 让AI理解现状（探索）
2. 让AI给出方案（评估）
3. 用户做出决策（决策）
4. 让AI执行（执行）
5. 用户验证结果（验收）

### 4.2 多Agent协作

当任务复杂时，主动要求多Agent并行：

```
"派发多个agent对项目进行一次全量的审查"
"使用合适的agent和skill进行一次全面的测试"
"使用朝廷工作流执行Task 1，务必遵守开发规范"
```
> 证据来源: C--Users-user 主会话

### 4.3 文档驱动

重度依赖文档来管理项目：

```
"仔细看看，应该有一份文档写了接下来的待办"
"issues/ISSUE-待办讨论结果-2026-06-04这个是对issues/ISSUE-DESIGN-未实现功能清单.md里面的内容讨论出来的结果"
"看看设计文档，还有几个细节没讨论完"
"把项目的文档整理一下，有点太分散了"
```
> 证据来源: C--Users-user 主会话

## 五、关键发现

### 5.1 "不信任但依赖" 的矛盾

- 大量保护性约束 → 不信任AI的自主判断
- 但完全依赖AI编码 → 自述"代码能力不强"
- 通过**严格的流程控制**来平衡这种矛盾

### 5.2 从"提问者"到"指挥官"的进化

**5月9日**（早期）:
```
"AI IDE是什么"
"Dify是什么"
"Orchestrator是什么"
```

**6月8日**（后期）:
```
"生成一份计划一个一个按顺序执行，改动要到位要能达到改动的目的，同时不要对其余正常功能产生影响"
"派发多个agent对项目进行一次全量的审查，包括代码bug审查，检查设计文档的功能是否真实实现"
```

30天内完成了从AI新手到AI指挥官的快速进化。

### 5.3 "质疑验证" 是核心习惯

用户从不盲目相信AI的输出：
- "真的无测试文件吗，我记得之前测试过啊"
- "验证之前的几个其他的agent给出的报告的真实性"
- "这是另一个ai搜索到的结果，你可以先核实一下信息的真实性再做判断"
- "agent是预定义的吧，你仔细看看"

### 5.4 "渐进式" 是执行原则

从不一次性下达所有任务：
- "逐条确认，一个一个来"
- "一次只讨论一个点"
- "1-3先改了，改完再把剩余的告诉我"

## 六、方法论总结

### 6.1 "先看后做" 原则
每个任务以"看看"开始，让AI先建立上下文理解再行动。

### 6.2 "效果前置" 决策法
执行前必问"能达到什么效果"，评估ROI再决定是否投入。

### 6.3 "保护性" 指令模式
使用"不要修改"、"只需要"等约束词，防止AI过度干预。

### 6.4 "渐进式" 执行策略
一步一步来，每步都可验证，出错可回滚。

### 6.5 "质疑验证" 习惯
对AI的结论保持怀疑，要求验证和证据。

### 6.6 "文档驱动" 管理
通过PROGRESS.md、ISSUE.md、design/等文档管理项目。
