## 一、协作模式概述

### 核心定位：AI-First 开发者

我是一名 AI-first 开发者，在 AgentHub 项目的开发过程中，AI 不仅是辅助工具，而是**核心开发伙伴**。整个项目的架构设计、代码实现、测试验证、问题排查都深度依赖 AI 协作。

## 二、六步协作法

### Step 1: 探索 — "看看项目"

每个任务开始前，我会让 AI 先了解现状，而不是直接动手。

**典型指令**：
- "D:\projects\agenthub"看看待办
- 看看设计文档，还有几个细节没讨论完
- 仔细看看这个网站的内容

**目的**：让 AI 建立上下文理解，避免盲目修改。

### Step 2: 评估 — "能达到什么效果"

在执行前，我会先问效果，评估 ROI 再决定是否投入。

**典型指令**：
- 改完能达到什么样的效果
- 改完能达到什么样的效果，告诉我效果就行，不要改代码
- 那方案一应该怎么实施呢，给我一份计划

**目的**：先评估再决策，避免做无用功。

### Step 3: 决策 — "做，方案是什么"

基于 AI 的分析和建议，我做出最终决策。

**典型指令**：
- 做，方案是什么
- 1-3先改了，改完再把剩余的告诉我
- 开始实现并保证改动达到真实目的且不影响其余正常功能

**目的**：我是决策者，AI 是执行者。

### Step 4: 执行 — "开始实现"

AI 按照计划执行，我设置约束条件防止越权。

**典型指令**：
- 不要修改任何文件只需要给我一份最终的报告
- 只需要测试并报告问题就行了，不要修改任何文件
- 只要测agenthub另外两个先不要管

**目的**：明确边界，防止 AI 过度干预。

### Step 5: 验收 — "测试验证"

我会验证 AI 的产出，确保达到预期效果。

**典型指令**：
- 测试一下opencode平台的适配情况
- 重启开发服务器后测试验证
- 验证之前的几个其他的agent给出的报告的真实性

**目的**：确保产出质量，不盲目相信 AI。

### Step 6: 迭代 — "还有什么待办"

完成一轮后，我会检查剩余工作，继续推进。

**典型指令**：
- 还剩些什么问题
- 再看看项目的所有文档还有哪些待办
- 把已完成的待办都标记为已完成

**目的**：持续迭代，直到项目完成。

## 三、关键协作习惯

### 3.1 "先别动手" 模式

在让 AI 执行任务前，先要求了解情况、讨论方案。

**实际案例**：
```
我: 怎么把页面换成现在的，要做些什么，先别开工，先告诉我
AI: [分析当前状态，给出改造方案]
我: 可以，你先做一版出来，我先看看
```

**价值**：避免盲目修改，确保方向正确。

### 3.2 "效果前置" 提问法

执行前必问效果，评估 ROI 再决定。

**实际案例**：
```
我: 改完能达到什么样的效果
AI: [分析改动的影响范围和预期效果]
我: 告诉我效果就行，不要改代码
```

**价值**：先评估再投入，避免做无用功。

### 3.3 "保护性" 指令模式

设置约束条件，防止 AI 越权。

**实际案例**：
```
我: "D:\projects\agenthub"派发多个agent对项目进行一次全量的审查，
    包括代码bug审查，检查设计文档的功能是否真实实现，有没有死代码写了但是完全没用到过，
    不要修改任何文件只需要给我一份最终的报告
```

**价值**：明确边界，保护现有代码。

### 3.4 "质疑验证" 习惯

对 AI 的结论保持怀疑，要求验证。

**实际案例**：
```
我: 真的无测试文件吗，我记得之前测试过啊，你先找找，真的没有就告诉我
我: 验证之前的几个其他的agent给出的报告的真实性
我: 这是另一个ai搜索到的结果，你可以先核实一下信息的真实性再做判断
```

**价值**：确保信息准确，避免被误导。

### 3.5 "渐进式" 执行策略

一步一步来，不一次性下达所有任务。

**实际案例**：
```
我: 逐条确认，一个一个来
我: 一次只讨论一个点
我: 1-3先改了，改完再把剩余的告诉我
```

**价值**：控制节奏，每步都可验证。

### 3.6 "文档驱动" 管理

通过文档管理项目，而非依赖记忆。

**实际案例**：
```
我: 以文件的形式记录在项目内
我: 不是写进记忆是写进文档
我: 加到设计决策文档里
我: 更新到文档里
```

**价值**：持久化知识，便于回顾和传承。

## 四、多 Agent 协作实践

### 4.1 并行审查

当任务复杂时，我会派发多个 Agent 并行工作。

**实际案例**：
```
我: "D:\projects\agenthub"派发多个agent对项目进行一次全量的审查，
    包括代码bug审查，检查设计文档的功能是否真实实现，有没有死代码写了但是完全没用到过
```

**效果**：多个 Agent 从不同角度审查，发现更多问题。

### 4.2 专项测试

针对特定功能，使用专门的 Agent 进行测试。

**实际案例**：
```
我: "D:\projects"了解一下项目情况，然后使用合适的agent和skill进行一次全面的测试，
    只需要测试并报告问题就行了，不要修改任何文件
我: 只要测agenthub另外两个先不要管
```

**效果**：聚焦测试范围，提高效率。

### 4.3 工作流驱动

使用自定义工作流（如"朝廷工作流"）规范开发流程。

**实际案例**：
```
我: 使用朝廷工作流执行Task 1，务必遵守开发规范
我: 使用朝廷工作流继续阶段 3
我: 继续使用朝廷工作流完成阶段 4，注意开发规范和验收规范，审查要更严格，这是核心部分
```

**效果**：规范化开发流程，确保质量。

## 七、协作理念总结

### 7.1 核心原则

1. **先看后做**：让 AI 先理解现状，再行动
2. **效果前置**：先评估 ROI，再决定是否投入
3. **保护性指令**：设置约束，防止越权
4. **质疑验证**：不盲目相信，要求证据
5. **渐进执行**：一步一步，可控可验
6. **文档驱动**：知识沉淀，便于传承

### 7.2 角色定位

- **我**：决策者、验收者、方向把控者
- **AI**：探索者、分析者、编码者、验证者

### 7.3 协作本质

> **我负责"想清楚"，AI 负责"做出来"。**
>
> 我的核心价值是：方向判断、需求把控、质量验收。
> AI 的核心价值是：信息收集、方案设计、代码实现。
